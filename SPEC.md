# Case Viewer v2 — Project Specification

## 1. Problem Statement and Project Purpose

Case Viewer is an informal analysis tool for volumetrics workflows, sitting between Petrel Output Sheet exports and formal DG documentation generated by tornadopy. Its job is mid-modelling diagnostic: paste in data, understand what changed, find what's driving the change, navigate across cases quickly.

The v1 implementation hit a maintainability wall. A single 284 KB index.html file, developed entirely via the GitHub web editor, reached a complexity level where adding features requires understanding 7,000+ lines of co-mingled HTML, CSS, and JavaScript. The commit history is a straight line of "Update index.html" — no diffs, no component isolation, no ability to test a change in isolation. Development stopped.

v2 is a complete rewrite with a proper module structure, addressing both the architectural problem and the feature gaps identified through analysis of actual use.

---

## 2. Scope and Positioning

### What this is
A browser-based, offline-capable analysis tool for exploring Petrel volumetric Output Sheet exports. Informal, fast, personal. The primary user is the Static Modelling Lead during active model iteration.

### What this is not
- A replacement for tornadopy (which produces formal DG plots from U&O Excel input)
- A shared/multi-user platform
- A Petrel plugin

### Relationship to the wider toolset
```
distribution-viewer   →   Petrel U&O workflow   →   tornadopy
(input distributions)     (run cases)               (formal DG output)
                                  ↓
                          Petrel Output Sheet
                                  ↓
                          case-viewer v2
                    (informal diagnostic, mid-modelling)
```

## 3. Architecture

### 3.1 Technology stack

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Framework | Vanilla JavaScript ES modules | No build step. Works from `file://` (with local server) or GitHub Pages |
| Styling | Tailwind CSS (CDN) | Consistent with existing tools |
| Charts | D3.js (CDN) | Deep integration for ball chart + cross-plot; already proven in v1 |
| Drag-reorder | Sortable.js (CDN) | Same as v1, lightweight |
| Icons | Font Awesome 6.4.0 (CDN) | Same as v1 |
| Persistence | localStorage | Consistent with v1 and distribution-viewer |

**Dropped from v1:** PapaParse — replaced by a custom parser that handles Petrel Output Sheet quirks directly.

**Why not React/Vue/Svelte:** The tool must work by opening index.html with a local server or deploying as a GitHub Pages static site. A build step adds friction that conflicts with the "just open the file" experience. ES modules with `<script type="module">` give component isolation without a bundler.

**`file://` limitation:** Browsers block ES module `import` on `file://` due to CORS. The development and deployment approach is:
- **Develop** with the modular file layout using `python3 -m http.server` during dev
- **Deploy** to GitHub Pages as-is (modules work over HTTP)
- **Optionally** provide a build script (`build.sh`: simple inline script that produces a single `index.html`) for true offline use

## 4. Data Model and Input

### 4.1 Supported input formats

All input is via **copy-paste** from Petrel Output Sheet into a textarea. No file drag-drop, no file picker.

#### Format 1: Standard table format (primary)
One row per zone/segment combination. Any number of numeric columns. This is the primary format that all views (pivot, ball, cross-plot, drivers) are built for.

```
Zone    Segment    Bulk volume [*10^3 m3]    Net volume [*10^3 m3]    STOIIP [1e6 Sm3]
North   Main       12500                      8750                     11.72
North   Flank      8200                       4920                     6.60
Central Main       9800                       6860                     7.44
```

**Detection:** First data row has >1 non-numeric cells (the zone/segment columns).

#### Format 2: Single line format, no grouping (accepted)
One row per case, field totals only. No zone/segment breakdown. Produces a single-row pivot table showing field totals.

```
Case    STOIIP [1e6 Sm3]    GIIP [1e9 Sm3]    HCPV oil [1e6 m3]
Case_322    49.65    2.14    215.34
```

**Detection:** Exactly 1 non-numeric cell (the case name) and plain property column names without parenthesised zone names.

**Use case:** Tracking a series of total-STOIIP revisions in the timeline view. User pastes one row at a time, each becoming a case.

#### Format 3: Single line format with grouping (rejected with helpful error)
One row per case, hierarchy encoded in column names like `STOIIP (North)`, `STOIIP (Central/Main)`.

```
Case    STOIIP (North) [1e6 Sm3]    STOIIP (Central) [1e6 Sm3]    STOIIP (Total) [1e6 Sm3]
Case_322    19.97    10.31    49.65
```

**Detection:** Exactly 1 non-numeric cell and column names containing parenthesised zone/segment names (pattern: `Property (GroupName)` or `Property (Zone/Segment)`).

**Behaviour:** Import modal shows a clear error:
> "This looks like a single line format export with zone/segment grouping. Please re-export from Petrel using the standard table format (uncheck *Single line format* in the Report tab)."

**Why reject:** The hierarchy is in column names rather than rows — a completely different parser path. Each case is a row, breaking the "paste one case" mental model. tornadopy already handles this format. The use case (comparing many realisations across one run) is tornadopy's domain.

### 4.2 Parser specification

The parser (`core/parser.js`) handles:

**Header processing:**
- BOM (UTF-8 with BOM) stripping
- Variable number of blank/header lines before the data — scan for the first line with tab-separated content
- Unit extraction from column headers: `STOIIP [1e6 Sm3]` → clean label `STOIIP`, unit `MCM`
- Scientific notation unit parsing: `[*10^3 m3]` → `KCM`, `[*10^6 m3]` → `MCM`, etc.
- Both comma and dot as decimal separators
- Auto-detection of an unlabeled first column (when data has one more column than headers, prepend `Zones`)

**Unit standardisation pipeline (`utils/units.js`):**

```
Raw unit string → standardizeUnit() → standard code (CM/KCM/MCM/BCM/TCM)
                                         ↓
                    divideBy1000 toggle → getScaledUnit() → display unit
                                         (KCM → MCM, MCM → BCM, etc.)
```

| Power | Standard code | Meaning |
|-------|---------------|---------|
| 10^0  | CM | Cubic metres |
| 10^3  | KCM | Thousand CM |
| 10^6  | MCM | Million CM |
| 10^9  | BCM | Billion CM |
| 10^12 | TCM | Trillion CM |

**Format detection (run before parsing data rows):**
1. Parse header line
2. Parse first data row
3. Count non-numeric cells in first data row
4. If >1 non-numeric → **Format 1** (standard table)
5. If exactly 1 non-numeric:
   - Scan column names for pattern `PropertyName (GroupName)` → **Format 3** (reject)
   - Otherwise → **Format 2** (single line totals)

**Data row processing:**
- Tab-split each row
- Numeric columns: parse as float, apply divide-by-1000 if toggled
- Non-numeric columns: keep as string (these become grouping keys)
- Rows with irregular column counts: surface a specific error with row number

**Auto-detect volume groups:**
- Identify which columns are likely zone/segment hierarchy keys: non-numeric, present in all rows, low cardinality relative to row count
- Pre-populate Volume Groups configuration in the import modal
- v1 reference list of known numeric columns: `Bulk volume, Net volume, Pore volume, HCPV oil, HCPV gas, STOIIP, GIIP`

### 4.3 Calculated parameters (`utils/parameters.js`)

Derived from raw volumetric values. Displayed when "Show Parameters" is toggled on in the pivot table.

```javascript
function calculateParameters(row, units) {
  // Unit-aware reads
  const bulkVolume = get('Bulk volume');
  const netVolume  = get('Net volume');
  const poreVolume = get('Pore volume');
  const hcpvOil    = get('HCPV oil');
  const hcpvGas    = get('HCPV gas');
  const stoiip     = get('STOIIP');
  const giip       = get('GIIP');

  return {
    GRV:   parseFloat(row['Bulk volume']),
    NTG:   bulkVolume > 0 ? netVolume / bulkVolume : 0,
    Por:   netVolume > 0  ? poreVolume / netVolume : 0,
    So:    poreVolume > 0 ? hcpvOil / poreVolume : 0,
    Sg:    poreVolume > 0 ? hcpvGas / poreVolume : 0,
    '1/Bo': hcpvOil > 0   ? stoiip / hcpvOil : 0,
    '1/Bg': hcpvGas > 0   ? giip / hcpvGas : 0,
    STOIIP: parseFloat(row['STOIIP']),
    GIIP:   parseFloat(row['GIIP']),
  };
}
```

Parameter display formatting:
- `NTG, Por, So, Sg` → percentage with 2 decimals (`× 100`, suffix `%`)
- `1/Bo, 1/Bg` → 2 decimal places
- `GRV, STOIIP, GIIP` → standard number formatting with 2 decimals

Parameters are calculated both at individual row level and as group/total summaries (sum raw volumes first, then derive ratios from the sums — not average of ratios).

## 5. Data Loading and Management Pipeline

### 5.1 Import flow

```
┌──────────────────────────────────────────────┐
│  IMPORT MODAL                                │
│                                              │
│  1. Paste data into textarea                 │
│  2. Format auto-detected, preview shown      │
│  3. Fill metadata: title, author, desc       │
│  4. Configure: divide/1000, volume groups    │
│  5. Confirm → case created                   │
└──────────────────────────────────────────────┘
```

**Step 1 — Paste:** User pastes tab-separated data from Petrel into a textarea. No file upload, no drag-drop.

**Step 2 — Detection and preview:**
- Parser runs on paste (debounced 300ms, same as v1)
- Format detected (Format 1, 2, or 3)
- If Format 3: show rejection error with re-export instructions. Stop here.
- If Format 1 or 2: show preview:
  - Detected columns listed (with units extracted)
  - First 3 data rows shown in a mini table
  - Volume Groups auto-populated from detected non-numeric columns
  - Row count shown

**Step 3 — Metadata:**
- **Field selector** — which field to add to (defaults to active field)
- **Case title** — free text, required
- **Author** — free text, with "Default" toggle (persists to localStorage)
- **Description** — optional textarea
- **Timestamp** — auto-set to now, clickable to reveal datetime picker for custom timestamp

**Step 4 — Import configuration:**
- **Divide by 1000** toggle — default on, persisted per case. Scales numeric values and adjusts unit display.
- **Volume Groups** — multi-level selector. Auto-populated from detected columns. User can add/remove levels.

**Step 5 — Confirm:** Case is created and becomes the active case. Data is parsed, stored, and visualised.

### 5.2 Post-import editing (CaseEditor.js)

All case metadata and import configuration can be edited after import. Accessible via:
- Edit button (pen icon) on case list hover
- Clicking the metadata card below the header

**Editable fields:**
- Case title (rename)
- Author
- Description
- Timestamp
- Volume Groups (re-configure hierarchy columns)
- Value Conversions (per discrete column):
  - Enable/disable toggle per column
  - Mapping table: original value → converted value
  - Copy button: copies conversion table to clipboard (tab-delimited)
  - Paste button: pastes conversion values from clipboard
  - Collapsible sections with persisted collapsed/expanded state

**Not editable after import:** The raw data itself and the divide-by-1000 setting. To change these, delete and re-import the case.

**On save:** Case data is updated in localStorage. Visualisations re-render. If value conversions changed, conversions are re-applied to the raw data.

### 5.3 Case management

- **Create** — via import modal
- **Edit** — via case editor (see 5.2)
- **Delete** — inline confirmation in case list (no browser `confirm()` dialog)
- **Duplicate** — copy a case with new name (for creating a variant with different value conversions)
- **Reorder** — drag-to-reorder in case list (Sortable.js). Order persisted per field.
- **Navigate** — prev/next buttons in header, arrow key shortcuts

### 5.4 Field management

- Create, rename, delete fields
- No default fields — users create their own from the browser
- Field selector: clickable field name in header (dropdown) + selector in settings panel
- Deleting a field deletes all its cases and associated settings
- Renaming a field migrates all localStorage keys

### 5.5 Persistence and storage

All data persisted in localStorage under versioned keys:
- `caseviewer_v2_state` — full app state (fields, cases, UI preferences)
- `caseviewer_v2_settings` — user preferences (default author, colour theme)

**Schema migration:** `state.schema` version field. On load, detect if stored schema < current, run migration function, re-save.

**Per-field settings persisted:**
- Selected metric
- Cross-plot X/Y metrics
- Legend layer selection
- Show legend toggle

**Per-case settings persisted:**
- Show values toggle
- Zoom enabled toggle
- Max depth slider value
- Extended settings panel open/closed
- Expanded zones state in pivot table

**Export/import (Phase 4):** JSON export of the full state or a single field. Import merges into existing state.

### 6.2 Pivot table (core view, significantly improved)

The pivot table is the central diagnostic surface. Shows the volumetric hierarchy for the active case.

**Column structure:**
- Hierarchy columns (Zone, Segment, Facies) — fixed left
- Volumetric columns — one per property, right-aligned

**Toggles:**
- **Hide empty rows** — rows where Bulk volume is 0
- **Show parameters** — switches display from raw volumes to calculated parameters (NTG, Por, So, Sg, 1/Bo, 1/Bg, GRV, STOIIP, GIIP)

**Collapsible rows:**
- Click Zone row to collapse/expand all Segments within it
- Expand/collapse all button
- State persisted per case per field

**Sorting:**
- Groups sorted by Bulk volume descending within each hierarchy level (retain v1 behaviour)
- Detail rows shown on expand at deepest level

**Totals:**
- Total row at bottom for each numeric column
- Sub-totals per Zone when expanded
- Parameters calculated from summed raw values (not averaged ratios)

**Legend:**
- Shows grouping hierarchy: "Level 1 = Zone, Level 2 = Segment"

**NEW: Delta mode**

When a Compare Case is selected, the pivot table gains delta columns alongside each volumetric:
```
Zone | Segment | STOIIP (Rev4) | STOIIP (Rev3) | Δ STOIIP | Δ%
```

- Δ coloured: green positive, red negative, grey zero
- Rows sorted by `|Δ|` in the active primary metric (toggle: sort by hierarchy vs sort by magnitude)
- Summary banner above table: total Δ STOIIP, top contributing zone/segment
- `Δ% = (new - old) / old × 100`
- Rows where both cases have zero are hidden in delta mode

### 6.3 Ball chart (retain from v1, improved)

D3 circle packing. Area proportional to selected metric. Coloured by zone (first hierarchy level).

**Retained from v1:**
- Metric selector (moved to persistent toolbar position in v2)
- Depth slider: controls max depth of hierarchy displayed
- Show values toggle: label each circle with its numeric value
- Enable zoom toggle: click-to-zoom interaction mode
- Show legend toggle: hide/show the circle legend
- Extended settings panel: collapsible panel with per-chart toggles
- Legend with layer slider (choose which hierarchy level the legend represents)
- Per-case settings persistence (show values, zoom, max depth, extended settings)
- Three-tier colour palette (light for level 1, medium for level 2, vibrant for level 3)
- Total metric display with unit annotation

**Improvements:**
- Legend: click to isolate a zone (grey out others)
- **NEW: Delta mode overlay** — when compare case active, circle border colour indicates direction of change (green border = grew, red = shrank), border width proportional to `|Δ%|`
- Export as SVG/PNG

### 6.4 Cross-plot (retain from v1, improved)

D3 scatter. Each point is a zone/segment combination. X/Y axes are independently selectable metrics.

**Retained from v1:**
- Axis labels clickable to change metric (dropdown with all volume + parameter metrics)
- Multi-case display: all loaded cases shown simultaneously with per-case symbols and colours
- Per-group and per-case visibility toggles in legend
- Grouping level selector: pick which hierarchy level (Zone vs Segment) for grouping
- Cross-plot settings persistence per field (X/Y metrics)
- Collapsible container (toggle button)

**Improvements:**
- Point labels: option to show Zone or Segment name on hover (tooltip) vs always
- Regression line toggle (simple linear, purely visual)

**Note:** Multi-case overlay was already in v1 — it is ported, not new.

### 6.5 NEW: Driver chart (informal tornado)

This is the primary new diagnostic view for answering "what's driving the change?"

**Trigger:** Select a compare case. The driver chart automatically becomes available.

**What it shows:** A horizontal bar chart where each bar is one zone/segment combination, sorted by `|Δ|` in the primary metric, descending. The bar extends left (negative) or right (positive) from a centre zero line. Bar length = `|Δ|`. Bar colour = positive/negative direction.

This is structurally identical to a tornado chart but derived from Output Sheet exports rather than U&O runs. It answers: "which part of my model explains this STOIIP change?"

**Controls:**
- Metric selector: which volumetric property to rank by
- Hierarchy depth: rank individual segments, or aggregate to zone level
- Filter: minimum `|Δ%|` threshold to exclude very small contributors
- Sort toggle: by absolute Δ vs by Δ%

**Reading the chart:** The top row is the zone/segment responsible for the largest absolute volume change. If one zone dominates the bar, that's where to look in the model.

**Export:** PNG. The chart title auto-includes: field name, compare pair (`Rev4 vs Rev3`), metric.

### 6.6 NEW: Revision timeline

A multi-line D3 line chart showing how key metrics have evolved across all loaded cases, ordered by revision number then timestamp.

**What it shows:**
- X axis: case timestamp or revision number
- Y axis: selected metric value
- One line per zone (up to 6-8 zones, coloured)
- Hover: vertical crosshair showing all zone values at that revision

**Use case:** "I've run 8 revisions this week — show me how STOIIP has tracked for the North zone across all of them." Also catches regressions: a line that dips and recovers flags something that was accidentally broken and fixed.

**Requirements:**
- Cases must have timestamps set (auto-populated from import time if not set manually)
- Cases from the same field are included
- Metric selector

### 6.7 Settings panel

The floating settings panel from v1 (gear icon, bottom-right) is retained but simplified. Settings per-field rather than per-session:
- Field selector
- Volume Groups configuration
- Value Conversions
- Default author

## 10. What Not to Port

- The monolithic `appState` object with ad-hoc property additions → replaced by structured state with schema versioning
- Inline event handlers scattered through HTML strings → module-scoped event handling
- The settings panel as the primary navigation surface → moved to header with view tabs
- localStorage keys without schema versioning → versioned keys with migration
- `id="pivot-body"` style DOM targeting of deeply nested elements → component-scoped rendering
- PapaParse dependency → custom parser
- `window._justAddedCase` / `window._addingGroupLevel` global flags → proper state management through event bus

## 11. Non-Goals (explicit)

- **No server.** The app runs from a local HTTP server or GitHub Pages. No Node.js backend.
- **No file upload.** Input is paste-only from Petrel. No drag-drop, no file picker.
- **No Python/tornadopy integration.** Browser-only. Reads Output Sheet exports; tornadopy reads U&O Excel.
- **No collaboration features.** Single-user, local. Share via JSON export if needed.
- **No Petrel plugin.** Data flows out of Petrel via the Output Sheet export, same as today.
- **No formal report generation.** That's tornadopy's job.
- **No single line format with grouping.** Detected and rejected with a helpful re-export message.
